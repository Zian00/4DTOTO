# Data Privacy and Security Considerations

This document describes privacy and security controls for the 4D/TOTO application and the remaining risks to track.

## 1. Data Collected

- Ticket image uploads (`/api/tickets/upload`, `/api/tickets/confirm`)
- OCR output from ticket images (`raw_ocr_text`)
- Ticket metadata:
  - game type
  - draw date / draw number
  - purchase date-time
  - selected numbers and bet details
- Result and notification records generated by the backend

## 2. Third-Party Processing

- OCR is performed via Google Gemini API.
- Uploaded ticket images are sent to Gemini for extraction.
- Implication:
  - Ticket content leaves local infrastructure and is processed by an external provider.
  - This should be disclosed clearly in user-facing documentation.

## 3. Security Controls Implemented

### Backend API

- File size limit:
  - Uploads are capped at 10 MB in `backend/routers/tickets.py`.
- File type validation:
  - Image magic-byte validation is enforced in `backend/utils/image_validation.py`.
  - Declared `content-type` must match detected content.
  - Allowed image MIME types are environment-configurable (`ALLOWED_IMAGE_MIME_TYPES`).
- Safer file persistence:
  - File extension is derived from validated MIME type, not raw client input (`backend/utils/storage.py`).
- CORS hardening:
  - CORS allow-list is environment-configurable (`CORS_ALLOW_ORIGINS`) in `backend/config.py`.
  - Wildcard CORS should only be used for local prototyping.
- Security headers:
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY`
  - `Referrer-Policy: no-referrer`
  - `Permissions-Policy: camera=(), microphone=(), geolocation=()`
  - Applied in `backend/main.py`.
- Reduced sensitive logging:
  - OCR raw text logging is disabled by default (`LOG_OCR_RAW_TEXT=false`).
- Upload cleanup:
  - On ticket deletion, image files are removed when no remaining ticket references the same file.

## 4. Privacy Controls and Data Minimization

- No user identity/profile system is currently implemented.
- Sensitive OCR text logging is suppressed by default.
- Only ticket-related data required for app features is stored.
- Recommendation:
  - Define retention periods (for example: auto-delete uploads after X days).
  - Add an admin tool/job to purge old uploads and stale OCR text.

## 5. Known Risks / Current Limitations

- No authentication/authorization yet:
  - Any client that can reach the API can access ticket/history endpoints.
- No per-user data isolation:
  - Ticket history is global without user accounts.
- Transport security depends on deployment:
  - HTTPS must be enforced by reverse proxy/platform in production.
- No encryption-at-rest strategy documented yet for DB/upload storage.
- No formal rate limiting / abuse controls yet.

## 6. Recommended Production Hardening

- Require auth for all ticket/notification endpoints (JWT or session-based).
- Implement per-user access controls on all queries.
- Enforce HTTPS and HSTS at ingress/reverse-proxy.
- Add request rate limiting and IP throttling.
- Add malware scanning for uploaded files.
- Encrypt DB volume and uploads volume at rest.
- Set explicit retention policy (images, OCR text, notifications, logs).
- Add audit logging for sensitive operations (upload/delete/read).
- Rotate API keys regularly and never commit secrets to git.

## 7. Environment Variables (Security Relevant)

- `GEMINI_API_KEY`
- `CORS_ALLOW_ORIGINS`
- `LOG_OCR_RAW_TEXT`
- `ALLOWED_IMAGE_MIME_TYPES`
- `UPLOAD_DIR`

## 8. User-Facing Disclosure (Suggested Text)

"Uploaded ticket images are processed by an external OCR provider (Google Gemini) to extract ticket fields. Predictions are educational features only and not gambling advice. Do not upload documents that contain unrelated personal or confidential information."
